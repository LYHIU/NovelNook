# 研究文档：个人书架管理

**日期**: 2026-06-11 | **关联**: [plan.md](./plan.md)

## 1. 晋江在线搜索方案

**决策**: 双通道策略 —— AJAX 快速通道（search_ajax.php）+ Baidu 搜索引擎回退

**理由**:
- 晋江 `search_ajax.php` 返回 JSON 结构化数据，解析零成本，响应 <2s
- 该接口限返回 5 条结果，覆盖面有限
- Baidu `site:jjwxc.net` 搜索作为补充，覆盖面更广，支持按作者筛选
- Baidu 回退仅在 AJAX 未命中且指定了作者名时触发，避免不必要的搜索引擎请求

**考虑过的替代方案**:
- 直接爬取晋江 HTML 搜索页 → 被反爬、HTML 结构频繁变更，已废弃（见 commit 1672ba3）
- 仅用 AJAX 通道 → 覆盖率不足，很多作品搜不到
- 自建索引库 → 过度工程化，不适合 MVP 阶段

## 2. 离线检测策略

**决策**: 使用 `navigator.onLine` + 网络请求失败回退的双重检测

**理由**:
- `navigator.onLine` 提供即时状态，但不是 100% 可靠（假阳性）
- 实际网络请求失败时降级为离线模式，展示友好提示
- Web App 和微信小程序均可使用各自的网络状态 API
- 纯本地操作（浏览书架、编辑笔记）完全不依赖网络，无需检测

**考虑过的替代方案**:
- 仅依赖请求失败检测 → 用户等待超时才知离线，体验差
- Service Worker 离线缓存 → 增加复杂度，MVP 阶段不需要

## 3. 本地存储方案

**决策**: Web App 使用 `localStorage`，微信小程序使用 `wx.setStorageSync`，键名 `private_bookshelf_v1`

**理由**:
- localStorage 配额 ≤5–10MB，单用户 500 本书约占用 0.5–1MB，远未触及上限
- 同步 API 确保数据立即持久化，无需处理异步竞态
- 微信小程序 Storage 上限 10MB/键，同样充足
- 无需 IndexedDB 的额外复杂度

**考虑过的替代方案**:
- IndexedDB → 更适合大数据量，当前规模不需要
- 服务端存储 → 违反宪法 IV 号原则（数据本地化），增加运维成本

## 4. JSON 导出/导入格式

**决策**: 导出一个 `.json` 文件，包含完整 `Book[]` 数组，导入时按 novelid 合并去重

**理由**:
- JSON 是人类可读格式，符合宪法"可读格式"要求
- novelid 是跨平台相对稳定的唯一标识（晋江作品 ID 不变）
- 合并策略（不覆盖）保护用户不会因导入旧备份而丢失最新数据
- 文件下载通过 `<a download>` / `Blob` 实现，无需后端参与

**考虑过的替代方案**:
- CSV 导出 → 不适合嵌套数据（标签数组、笔记多行文本）
- 覆盖式导入 → 用户可能丢失较新的手动修改
- 云端同步 → 超出 MVP 范围

## 5. 非晋江平台策略

**决策**: MVP 阶段仅晋江支持在线检索，其他平台（长佩、番茄、起点）生成官方搜索链接

**理由**:
- 晋江已有稳定可用的 JSON API（search_ajax.php）
- 其他平台缺乏同等级别的开放接口，HTML 抓取风险高且维护成本大
- 宪法合规边界要求"优先使用官方 API"
- 后续版本按平台优先级逐步接入

**考虑过的替代方案**:
- 一次性接入四个平台 → MVP 范围膨胀，且其他平台的技术方案尚不明确
- 不提供任何平台在线检索 → 违背用户核心需求"检索到书籍信息"
